# ADR-001: Выделяем сервис "Вакансий и тестирования кандидатов"

## Status

- _Proposed_

## Context

На основе требований и консёрнов стейкхолдеров, были выписаны характеристики системы, которые затем были соотнесены с различными поддоменами и их bounded контекстами:

Текст требования **[US-081]**:
> Мы ожидаем 1 к заявок в день от рандомных котов, также, судя по отзывам, наши конкуренты могут попытаться нас заддосить в этом месте. Они так делали уже несколько раз с другими компаниями, после чего компании закрывались с позором.

Консёрн, который был озвучен нам CEO компании:
> релизный цикл для всей системы — месяц, для скоринга работников — неделя максимум.

Исходя из анализа архитектурных характеристик, было выявлено, что к данному bounded context предъявляются более строгие требования к reliability, scalability, elasticity, чем к остальным частям системы, а также более строгий релизный цикл - 1 неделя против 1 месяца для остальной системы.  
Необходимо найти такое архитектурное решение, которое оптимальным образом позволит удовлетворить требования, выставляемые к системе, без излишнего переусложнения остальных частей системы.

## Decision

Мы выделим bounded context "Размещение вакансий и тестирование кандидатов" в отдельный сервис, со своей отдельной базой данных.  
Это решение позволит сфокусировать усилия по скалированию и обеспечению высокой надежности только в рамках одного сервиса, не переусложняя и не удорожая инфраструктуру остальной системы.  
Также это позволит более гранулярно контролировать релизный цикл, не выставляя жестких требований по релизам другим командам, поддерживающим другие сервисы системы.  

Если этого не сделать - то придется обеспечивать те же значения характеристик availability, reliability, scalability, elasticity для всех контекстов в рамках одного и того же сервиса, что может оказаться дороже по инфраструктуре и стоимости разработки.

## Consequences

Как следствие этого решения, для обеспечения более высоких значений характеристик availability / fault tolerance для всей системы, мы будем использовать асинхронный неблокирующий вид коммуникации через message broker, чтобы получить данные о воркерах в других сервисах ("Матчинг", "Маркетплейс услуг" и "Биллинг воркеров").  
Выбор message broker - ожидайте в других ADR (и в рамках другого курса, лол).
